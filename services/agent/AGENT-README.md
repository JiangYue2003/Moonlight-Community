# AGENT README：把“AI 调用”升级成“个人知识助手”的完整设计说明

这份文档不是代码注释，也不是汇报材料。  
它更像一篇“从 0 到 1”复盘：讲清楚这个 AGENT 为什么这样设计、已经做到了什么、以及每一块技术背后的取舍。

---

## 1. 先说目标：这个 AGENT 到底在解决什么问题

在这个项目里，AGENT 的目标不是“回答一切问题”，而是更务实的一件事：

1. 只围绕用户自己的收藏知文，构建个人知识底座。
2. 让用户可以像聊天一样提问，并得到带引用来源的回答。
3. 在不影响主业务性能的前提下，把“检索 + 推理 + 记忆”做成可持续演进的能力。

这决定了它的基本形态：

1. 它是独立模块，不侵入主链路。
2. 它是“检索增强问答（RAG）优先”，不是纯大模型自由发挥。
3. 它是“可解释优先”，回答要能回溯到具体内容片段。

---

## 2. 总体架构：为什么拆成两个服务

我们把 AGENT 分成了两个运行面：

1. `agent-api`：在线对话面
2. `agent-indexer`：异步入库面

这样拆分的原因很直接：

1. 在线问答要快，不能被切块、向量化这类重计算拖慢。
2. 收藏同步、索引构建天然是后台任务，适合事件驱动和重试补偿。
3. 出故障时可以隔离，索引链路波动不应该拖垮聊天接口。

一句话概括：用户提问只读“已准备好的知识”，而不是临时现做。

---

## 3. 为什么选 ReAct，而不是 Reflection / Plan-and-Execute

这是一个典型的工程取舍问题。

### 3.1 ReAct 适合当前阶段

ReAct（Thought -> Action -> Observation）最大的优势是：

1. 能显式调用工具（检索、记忆读取、融合等），与 RAG 天然匹配。
2. 推理链路可控，便于加入超时、步数、熔断。
3. 对网页问答场景响应更快，用户体感好。

当前项目采用的是 **ReAct-lite**：限制步数、限制工具预算、限制超时，并在复杂问题上支持后端隐藏式“计划-检索-校验”多轮回路，以稳定性优先。

### 3.2 为什么暂不优先 Reflection

Reflection 的核心价值在“自我反思和多轮修正”，通常能提升复杂任务质量。  
但它带来的现实成本也明显：

1. token 成本更高
2. 延迟更长
3. 工程控制复杂度更高（尤其是线上稳定性）

对“个人知识问答”这个主场景来说，先把召回质量和引用可靠性打牢，收益更大。

### 3.3 为什么暂不优先 Plan-and-Execute

Plan-and-Execute 更适合长流程、多工具编排的复杂任务（如自动化工作流）。  
当前 AGENT 的主要路径还是“检索 -> 生成 -> 引用”，并非重 workflow 场景。

所以阶段性结论是：

1. 现在：ReAct-lite 最合适
2. 未来：当工具生态变复杂，再引入 Plan-and-Execute 作为进阶模式

## 4. RAG 主链路：从“用户问题”到“可引用答案”

在线问答不是把问题直接丢给模型，而是先“找资料”再“组织回答”。

主链路可以理解为六步：

1. 识别意图：这更像精确查询、概念问题，还是关系问题。
2. 多路召回：并行走语义检索、关键词检索、记忆检索（关系通道可按开关启用）。
3. 结果融合：用 RRF 把多路结果拉到同一排序空间。
4. 组装上下文：构造成“只基于这些上下文作答”的提示语境。
5. 模型路由：按复杂度在 Lite/Pro 模型间选择。
6. 流式输出：SSE 实时回传 token，并附带 citation 与 final 事件。
7. 隐藏式 Planner：复杂问题先内部拆解为 1-3 个子查询，再做分治检索与证据压缩；前端事件不增加。
8. 隐藏式 ReAct 回路：复杂问题可进入有限步数的检索-观察-校验闭环，必要时改写 query 再检索，并带有循环检测与超时保护。

这里的关键点是：模型不直接“想答案”，而是先“看证据”。

---

## 5. 多路召回：为什么不是单一向量检索

单一向量检索对“语义相近”很强，但对编号、术语、精确字段匹配会有短板。  
所以当前设计采用多路并行：

1. 向量通道（Milvus / ES 向量）：解决语义相似问题。
2. BM25 通道（Elasticsearch）：解决关键词、术语、编号匹配。
3. 记忆通道（长期记忆库）：补充用户显式固化的事实与回答偏好。
4. 关系通道（图谱）：用于实体关系问题，当前按 feature flag 控制。

融合策略使用 **RRF（Reciprocal Rank Fusion）**，核心逻辑是“每路排名都有贡献，避免被单一路径垄断”。  
这对复杂问题特别实用：既不丢语义，也不丢精确命中。

### 5.1 可优化项：引入 Cross-Encoder Reranker（精排层）

当前系统在多路召回后使用 RRF 融合并直接截断 TopK，已经能稳定工作。  
进一步提升质量时，最值得优先加入的是 **Cross-Encoder Reranker**，思路是：

1. 先用现有多路召回拿到候选集（例如 TopN=40）。
2. 将每个候选与问题组成一对输入（Query + Passage）。
3. 让 Cross-Encoder 对每一对直接输出“相关性分数”。
4. 按该分数重排，再取 TopM（例如 8-12）进入生成。

它比“仅向量相似度排序”更强的原因在于：  
Cross-Encoder 是“联合编码”，问题 token 与片段 token 在同一模型里做细粒度交互，因此对否定、约束、指代、条件语义更敏感，通常能显著提升最终答案命中率与引用准确性。

工程上可以把它放在 RRF 之后，作为“二阶段精排”：

1. 第一阶段追求召回率（Recall）
2. 第二阶段追求排序精度（Precision）

代价是推理成本和延迟上升，所以建议采用“预算化启用”：

1. 仅在复杂问题或低置信结果时启用
2. 仅对 TopN 小窗口做精排，控制时延
3. 配合熔断与降级，异常时回退到 RRF 结果

## 6. 向量数据库与搜索系统：为什么是 Milvus + Elasticsearch

这不是“二选一”，而是职责分工：

1. **Milvus**：做语义向量检索主力，适合高维向量 ANN 场景。
2. **Elasticsearch**：做 BM25、结构化过滤，同时可承接向量镜像能力。

这么做的实际好处：

1. 性能层面：语义和关键词各走最擅长的引擎。
2. 工程层面：ES 同时承担元数据检索与可观测运营查询。
3. 风险层面：当某一路异常时，可以降级到另一条路径保服务。

## 7. 收藏内容如何变成“可检索知识”

这部分由 `agent-indexer` 异步完成，核心是“事件驱动 + 幂等 + 可补偿”。

### 7.1 入库触发

当用户收藏知文后，索引服务消费收藏事件，拉取正文并开始处理。

### 7.2 文本处理与切块策略

切块不是简单按长度硬切，采用“双层策略”：

1. 先按标题/章节语义边界分段，保留结构信息。
2. 再按窗口长度切块，并设置 overlap，降低上下文断裂。

每个 chunk 会携带关键元数据（user_id、post_id、chunk_id、version 等），保证可追溯和可过滤。

如果你把它理解成“把一整篇文章打碎再检索”，这个理解是对的，但关键在于“怎么打碎”。

### 7.2.1 清洗：先把文本变成“可检索文本”

在当前实现中，知文正文不会做激进改写，主要做的是“结构保留 + 格式归一”，目标是减少噪声、保住语义：

1. 结构识别：优先按 Markdown 标题（`# / ## / ### ...`）识别章节边界。
2. 空白归一：连续空白、制表、全角空格等会被折叠，避免 embedding 受格式噪声影响。
3. 字符归一：通过 NFKC 归一化处理全角/半角、兼容字形，降低同词异形带来的召回偏差。
4. 截断保护：在部分文本后处理场景会按 rune 截断，避免超长字段污染索引。

这一步的本质不是“美化文本”，而是让后续切块和向量化更稳定。

### 7.2.2 切块：为什么是“先按标题，再按窗口”

当前切块分两层：

1. 第一层：按标题切段  
2. 第二层：段内滑窗切块（`size=800`, `overlap=100`，按 rune 计，不按 byte）

为什么这么设计：

1. 先按标题切段：保留主题边界，避免把“不同章节观点”混进同一块。
2. 再按窗口切块：控制每块长度，适配 embedding 模型的上下文容量。
3. 加 overlap：让相邻块共享一部分上下文，减少“关键信息卡在块边界”造成的召回损失。
4. 按 rune 切分：对中文更安全，不会切坏字符边界。

直观理解：  
`size=800` 像是每个知识片段的“最大篇幅”，`overlap=100` 是片段之间的“衔接区”。

### 7.2.3 到底对什么内容做了向量化

这里容易误解。并不是只把 `user_id/chunk_id` 这类字段做向量化。  
真正向量化的是“可读文本内容”，主要有两类：

1. 知文内容块（知识库向量）
2. 长期记忆事实文本（由 `subject + predicate + object` 拼接出的事实句）

而 `user_id/post_id/chunk_id/version/status` 这些字段是检索过滤与追溯字段，不参与语义向量本身。

也就是说：

1. 语义相似由“文本向量”决定
2. 权限隔离与可追溯由“结构化字段”保证

两者是分工，不是二选一。

### 7.3 向量化与双写

切块后进行 embedding，随后并行写入：

1. Elasticsearch（关键词与索引管理）
2. Milvus（语义向量检索）

这里的“并行双写”有一个很实际的好处：  
即使向量通道临时抖动，关键词通道仍可服务；反过来也一样。在线问答可以降级，但不至于整体不可用。

状态按 `pending/indexing/ready/failed` 管理，在线检索只读 `ready`。

### 7.4 更新与补偿

为保证新增收藏能快速生效，索引链路配有：

1. 幂等键去重
2. 失败重试与退避
3. 收藏真值对账（避免“已取消收藏但残留索引”）

## 8. SSE 流式传输：为什么前端体验更像“在思考”

问答接口采用 SSE，不是一次性返回整段文本。  
前端会依次收到多个事件：

1. `tool_call`：开始检索
2. `tool_result`：检索完成与引用计数
3. `route`：模型路由决策（可选暴露）
4. `token`：逐段回答内容
5. `citation`：引用来源
6. `final`：最终答案
7. `[DONE]`：流结束

这样做的价值：

1. 降低首字等待焦虑
2. 可视化工具链路，增强可解释性
3. 便于前端在失败场景做优雅降级

## 9. 记忆系统：短期记忆与长期记忆各自做什么

### 9.1 短期记忆（Redis）

短期记忆服务于“当前会话连续性”，通常包括：

1. 最近对话窗口
2. 会话摘要
3. 上下文 TTL 生命周期

它强调的是速度和上下文粘性，不强调永久保存。

当前实现不是抽象概念，而是明确的“窗口 + 摘要”算法：

1. 对话写入：每轮问答结束后，把 `user` 和 `assistant` 两条消息写入 Redis List。
2. 窗口截断：固定保留最近 `24` 条消息（约 12 轮问答），超过即裁剪旧消息。
3. 摘要触发：当消息数达到窗口上限时，生成会话摘要并单独写入 summary key。
4. 摘要算法（当前版本）：取窗口内最后一条消息，生成“会话摘要：最近讨论了 xxx”的轻量摘要。
5. TTL 策略：
6. 消息窗口 TTL 默认 `24h`（`SessionTTLHours`）。
7. 摘要 TTL 默认 `72h`（`SummaryTTLHours`）。

这说明短期记忆当前是“工程可控优先”的版本：先保证稳定上下文连续性，再逐步升级摘要质量。

### 9.2 长期记忆（MySQL + Milvus）

长期记忆服务于“跨会话稳定事实与偏好”，通过 `memory/pin` 等方式沉淀。  
处理思路是：

1. 抽取结构化事实
2. 做置信度过滤与去重合并
3. 结构化事实落 MySQL
4. 语义表示落 Milvus

在线问答时可把长期记忆作为独立召回通道，补充用户个性化背景。

当前长期记忆采集与入库规则如下：

1. 采集入口：`memory/pin` 接口（用户显式“固定记忆”）。
2. 抽取方式：LLM 从 pin 内容中抽取事实三元组 JSON。
3. 单条事实字段：
4. `subject`
5. `predicate`
6. `object`
7. `confidence`（0~1）
8. `sourceRef`
9. 过滤规则：`subject/predicate/object` 任何一项为空则丢弃，`confidence < 0.5` 丢弃。
10. 事实唯一标识：`fact_id = sha1(user_id|subject|predicate|object)`，用于幂等更新。
11. MySQL 存储字段（结构化层）：
12. `user_id, fact_id, subject, predicate, object_value, source_ref, confidence, version, status, created_at, updated_at`
13. 以 upsert 方式写入，状态默认 `active`。
14. Milvus 存储字段（语义层）：
15. 将 `subject + predicate + object_value` 拼接后做 embedding。
16. 向量记录包含 `user_id, fact_id, subject, predicate, object_value, source_ref, version, status, confidence, vector`。
17. 检索隔离：无论 MySQL 检索还是 Milvus 检索，都强制按 `user_id` 过滤，只返回当前用户事实。

此外，系统新增了“偏好记忆”这一类软记忆，用来沉淀用户偏好的回答方式和关注点，例如：

1. 更关心代码实现，而不是产品描述
2. 回答偏扁平叙述，不要太多分点
3. 当前主要关注 feed / cache / agent 模块

首版偏好记忆特点：

1. 仍复用 `memory/pin` 入口，不新增对外 API。
2. 在异步抽取时，与事实抽取并行进行，但单独落表，不混入事实三元组。
3. 仅保存高置信、稳定的长期偏好，不保存“这次先简单点”这类一次性要求。
4. 在线回答时把高置信偏好转成 system instruction，影响回答风格，但不改变“必须基于证据回答”的硬约束。

换句话说，长期记忆在当前实现里不再只是“事实卡片化 + 双存储（结构化 + 向量化）”，还增加了一层“软偏好约束记忆”。

## 10. 提示词工程：如何让模型“少幻觉、重证据”

这个 AGENT 的提示词策略不是追求“华丽表达”，而是约束回答边界：

1. 明确角色：个人知识助手。
2. 明确规则：只基于给定上下文回答，不编造。
3. 明确输出倾向：简洁、中文、可引用。
4. 明确兜底：无命中时直说“不确定”，并提示补充收藏信息。

结合 RAG，上下文是“证据”，提示词是“行为约束”，两者一起减少幻觉。

---

## 11. 稳定性设计：模型路由、回退、熔断

为了成本与效果平衡，系统实现了多模型路由：

1. 默认 Lite，复杂问题升级 Pro。
2. 判断维度包括问题长度、上下文长度、召回规模、会话复杂度等。
3. Pro 异常时支持自动回退 Lite，保证服务连续性。
4. 结合 Redis 失败窗口做软熔断，避免持续打到不健康模型。

这套机制的本质是：把“模型选择”变成可观测、可调参、可降级的工程能力。

---

## 12. 安全与沙箱：防止 AGENT 越权读取信息

AGENT 的最大风险不是“答错”，而是“拿到不该拿的数据”。  
因此安全策略是硬约束而非建议：

1. 工具白名单：未注册工具不可调用。
2. 严格参数校验：类型、长度、阈值、枚举必须通过校验。
3. 用户隔离：所有检索必须强制 `user_id` 过滤。
4. 审计留痕：工具调用全链路记录，便于回放与追责。
5. 默认无文件系统读取能力、无任意命令执行能力。
6. 注入防护：对可疑提示注入模式做输入拦截与策略隔离。

目标很明确：LLM 可以“推理”，但不能“越权”。

## 13. 与其他微服务如何协同且不互相拖累

协同方式遵循“在线轻依赖、离线重处理”的原则：

1. 鉴权通过 auth 中间件完成，确保会话归属正确。
2. 收藏内容处理走异步索引链路，不阻塞主业务请求。
3. 收藏真值校验用于补偿环节，避免索引脏数据。
4. AGENT 使用独立连接池、限流、资源预算，控制性能边界。

这样可以保证：AGENT 增强能力在增长，但主业务稳定性不被牺牲。

## 14. 可观测性：不是“有日志”，而是“可度量”

当前已具备模型路由与调用链路的指标化能力，包含：

1. 路由命中统计
2. 回退次数
3. Pro 熔断触发
4. 调用耗时
5. token 用量
6. 估算/实际成本

这些指标让我们能回答三个核心问题：

1. 质量是否在提升
2. 成本是否可控
3. 异常是否能被快速定位

## 15. 现在已经具备什么能力

截至当前阶段，这个个人知识助手已经具备：

1. 会话创建、历史查询、SSE 流式问答
2. 多路召回（语义 + 关键词 + 记忆）与 RRF 融合
3. 引用返回与可追溯回答
4. 收藏驱动的异步知识入库与补偿
5. 短期记忆与长期记忆固化入口
6. 多模型路由、回退、软熔断
7. 基础安全与可观测体系

它已经不是一个“单次 AI 调用接口”，而是一个有运行边界、有知识底座、有治理能力的 AGENT 模块。

## 16. 下一步可以怎么演进（可选）

如果继续升级，优先级建议是：

1. 引入 Cross-Encoder reranker 作为 RRF 后二阶段精排（提升相关性排序精度）
2. 完成图谱通道实装（关系问题显著增益）
3. 强化记忆生命周期策略（冲突消解、衰减、压缩）
4. 细化安全策略（PII 脱敏、风险分级、策略引擎）
5. 视复杂任务增长，再考虑多 AGENT 编排

先把“单 AGENT + 强工具链”打磨扎实，通常比过早拆多 AGENT 更稳。
