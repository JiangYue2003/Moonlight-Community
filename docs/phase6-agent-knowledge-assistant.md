# Phase 6 - 个人知识助手 AGENT 实施蓝图（中文）

最后更新：2026-05-17  
适用范围：`zhiguang-go` 新增独立 AGENT 模块（网页个人知识助手）

## 1. 目标与硬约束

本阶段目标：
1. 基于用户“已收藏知文”构建个人知识底座。
2. 提供可追溯引用的对话式问答助手。
3. 逐步演进为知识图谱增强的个人助手。

硬约束（必须满足）：
1. AGENT 必须作为独立模块/独立服务运行。
2. 不影响现有主链路生产可用性与性能（`knowpost/search/llm/relation/auth`）。
3. 重计算流程（清洗、切块、向量化、重排）必须异步化，不能挂在主请求链路上。

## 2. 模块边界与部署策略

### 2.1 新增模块
1. `services/agent/api`
   1. 对外提供知识助手接口（HTTP + SSE）。
   2. 承载 AGENT 编排、工具调度、路由、记忆读写。
2. `services/agent/indexer`
   1. 异步处理收藏知文入库：清洗、切块、向量化、索引写入。
3. `services/agent/shared`
   1. 共享协议与组件：召回融合、工具契约、路由规则、安全策略。

### 2.2 零侵入主链路原则
1. 现有 API 不同步调用 `agent/*`，避免主链路被 AGENT 放大延迟。
2. 收藏内容接入优先采用异步方式：
   1. 事件流消费（推荐）
   2. 定时增量拉取（备选）
3. AGENT 使用独立资源池：
   1. 独立连接池（MySQL/Redis/ES/Milvus/Neo4j）
   2. 独立限流与熔断配置
   3. 独立副本与弹性策略

### 2.3 资源预算（防性能挤占）
1. 请求级预算：
   1. AGENT 全链路超时
   2. 单工具超时
   3. 最大推理步数
   4. 最大召回/重排候选量
2. 服务级预算：
   1. 并发上限
   2. 每用户 QPS 上限
   3. 每请求 token 成本上限

## 3. 多路召回与知识处理流水线

## 3.1 一期在线召回（双通道）
1. 语义通道：Milvus ANN，`top_k=20`。
2. 关键词通道：Elasticsearch BM25，`top_k=20`。
3. 融合：RRF（Reciprocal Rank Fusion），常量 `k=60`。
4. 重排：融合 TopN（如 40）进入 rerank，产出 TopM（如 8~12）供生成。

## 3.2 二期图谱召回（关系通道）
1. Neo4j 用于实体关系问题（例如“谁负责哪些项目”）。
2. 一期先保留工具接口 `search_graph`（默认关闭）。
3. 二期通过 feature flag 启用。

## 3.3 收藏知文入库流水线（异步）
输入：用户收藏知文。  
流程：
1. 获取内容与元数据（只读）。
2. 文本规范化（Markdown 清洗、标题路径保留、去噪）。
3. 双层切块：
   1. 按标题/段落切分
   2. 按长度窗口二次切分并设置重叠
4. 向量化（embedding）。
5. 双写并行索引：
   1. Milvus（语义主检索）
   2. ES（BM25 + 可选向量镜像）
6. MySQL 记录 chunk 元数据（版本、哈希、来源、所属用户）。

## 3.4 必备元数据字段
每个 chunk 至少包含：
1. `user_id`
2. `post_id`
3. `chunk_id`
4. `title_path`
5. `tags`
6. `publish_time/create_time`
7. `version/hash`
8. `entity_refs`（二期可选）

默认检索必须按 `user_id` 强过滤。

## 3.5 收藏增量同步与 1-3 分钟可检索 SLA

目标：用户新增收藏后，知识助手在 1-3 分钟内可检索到该内容。

1. 主路径（事件驱动，推荐）
   1. 收藏动作写入 outbox/event（不阻塞主请求）。
   2. `agent-indexer` 消费事件并进入处理队列。
   3. 执行清洗、切块、embedding、ES/Milvus/MySQL 双写。
   4. 索引状态推进：`pending -> indexing -> ready`。

2. 状态机与可见性
   1. `pending`：已记录待处理。
   2. `indexing`：处理中，不参与在线召回。
   3. `ready`：可检索并可被引用。
   4. `failed`：处理失败，等待补偿重试。
   5. AGENT 在线召回默认仅使用 `ready` 数据。

3. 时效指标（SLA）
   1. P50：<= 60 秒。
   2. P95：<= 180 秒。
   3. 超过 P95 的样本进入补偿队列并触发告警。

4. 补偿与幂等
   1. 每 10 分钟执行一次“收藏-索引状态”增量对账任务。
   2. 对 `failed` 或超时未完成项执行指数退避重试。
   3. 以 `user_id + post_id + version/hash` 作为幂等键，避免重复写入。

5. 运行保障
   1. `agent-indexer` 与主业务 worker 独立消费组、独立资源配额。
   2. 单用户任务并发上限可控，防止热点用户拖垮队列。
   3. 索引链路支持熔断降级（仅 ES 或仅标记待重试）。

## 4. AGENT 运行时设计（EINO + ReAct-lite）

## 4.1 推理范式
默认采用 ReAct-lite：
1. Thought
2. Action（调用工具）
3. Observation
4. Final

约束：
1. 最大 3~4 步。
2. 每步有独立超时。
3. 达到步数/超时立即降级返回。

## 4.2 工具系统（Go interface 插件化）
最小工具集合：
1. `get_user_favorites`
2. `search_vector_milvus`
3. `search_vector_es`
4. `search_bm25_es`
5. `fuse_rrf`
6. `rerank`
7. `memory_read`
8. `memory_write`
9. `search_graph`（二期启用）

## 4.3 查询路由（意图识别）
规则优先、模型兜底：
1. 含编号/代码/精确匹配：提升关键词通道权重。
2. 概念解释类：提升语义通道权重。
3. 关系类：二期启用图谱通道权重。

路由输出“通道权重”，而不是单通道硬切。

## 4.4 稳定性控制
1. 三态熔断（Closed/Open/HalfOpen）覆盖：
   1. LLM 调用
   2. 外部检索调用
2. 降级策略：
   1. Milvus 不可用 -> BM25-only
   2. rerank 不可用 -> 直接使用融合结果
   3. LLM 高负载 -> 模板化短回答 + 引用片段

## 4.5 多模型路由
1. 简单问题 -> 低成本模型
2. 多步推理问题 -> 高能力模型
3. 决策因子：
   1. 问题复杂度
   2. 上下文长度
   3. 预估工具调用数
   4. 成本预算

## 5. 记忆系统设计

## 5.1 短期记忆（Redis）
1. 会话窗口上下文
2. 最近轮次摘要
3. TTL 管理生命周期

## 5.2 长期记忆（MySQL + Milvus）
1. MySQL：结构化偏好、学习目标、显式记忆
2. Milvus：长期语义记忆检索
3. 写入尽量异步，不阻塞主回答路径

## 5.3 记忆策略细化（执行级默认值）

1. 短期记忆模型：热窗口 + 会话摘要
   1. 热窗口保留最近 `N=12` 轮原始对话（可配置范围 8-20）。
   2. 超出窗口的历史归并为“会话摘要块”。
   3. 归并触发条件：
      1. 累积 6 轮对话；或
      2. 上下文 token 超过预算阈值。

2. 短期记忆存储与 TTL
   1. 热窗口（Redis）：TTL `24h`。
   2. 会话摘要（Redis）：TTL `72h`。
   3. 空闲会话自然过期，长期价值信息再下沉到长期记忆。

3. 短期记忆字段（最小集）
   1. 用户问题
   2. 助手回答
   3. 工具结果摘要（非原始全量）
   4. 引用 `post_id/chunk_id`
   5. 会话目标（当前任务）
   6. 未决问题列表

4. 写长期记忆触发条件
   1. 用户显式 `memory/pin`。
   2. 高频出现且高置信的稳定事实（规则阈值触发）。
   3. 会话结束后的异步归档任务。

## 5.4 长期记忆压缩与可追溯策略

采用“双层压缩”，兼顾检索效率与事实可追溯：

1. 事实层（MySQL，保真）
   1. 形态：事实卡片/三元组（subject-predicate-object/value）。
   2. 每条事实必须带来源：`post_id/chunk_id/version/hash/time`。
   3. 最小字段建议：
      1. `user_id`
      2. `fact_id`
      3. `subject`
      4. `predicate`
      5. `object_or_value`
      6. `source_ref`
      7. `confidence`
      8. `version`
      9. `created_at`

2. 语义层（Milvus，检索）
   1. 对“事实摘要 + 段落摘要”生成向量并入库。
   2. 在线查询先语义召回，再回查事实层做校验补全。
   3. 最终回答必须可映射到事实来源引用。

3. 压缩与更新流程
   1. 抽取事实 -> 去重归并 -> 生成摘要 -> 向量化。
   2. 文档更新时按 `version/hash` 做增量重建。
   3. 旧版本采用软失效，避免检索脏读。

## 7. LLM 工具调用约束、沙箱与安全（重点）

本节为强制执行项，目标是确保 LLM/AGENT 不能读取或推断不应访问的信息。

## 7.1 权限模型（默认拒绝）
1. 工具调用采用白名单模型：未注册工具不可调用。
2. 每个工具必须绑定权限标签（read_scope/write_scope/admin_scope）。
3. 默认只开放“只读工具”，写工具需显式开启并二次确认。
4. 工具调用时必须注入并校验 `user_id`，严禁模型自由指定越权用户。

## 7.2 数据隔离与行级约束
1. 所有检索工具强制带 `user_id` 过滤条件。
2. 禁止返回跨用户文档引用；后置结果再做一次 ownership 校验。
3. 长期记忆读取必须按 `user_id + tenant_id` 双键约束（如存在多租户）。
4. 禁止“无过滤全表检索工具”暴露给 LLM。
5. 记忆写入必须绑定 `user_id`，禁止跨用户归并。
6. 收藏增量任务按用户分片执行，防止跨用户污染。

## 7.3 工具参数约束与校验
1. 每个工具定义严格 schema（类型、长度、枚举、正则）。
2. 关键参数做阈值保护（TopK、时间范围、批量大小）。
3. 文本输入做注入防护（提示注入关键词、SQL/Cypher 片段拦截与转义）。
4. LLM 生成的工具参数必须经过服务端验证后才能执行。

## 7.4 执行沙箱与网络边界
1. AGENT 运行节点禁止任意外网访问（仅允许白名单服务域名/IP）。
2. 禁止执行系统命令类工具；如未来需要，必须运行在隔离沙箱容器并只读文件系统。
3. 工具运行使用最小权限账号（ES/Milvus/Neo4j 分角色只读）。
4. 对 Neo4j/Cypher 使用参数化查询，禁止字符串拼接执行。

## 7.5 Prompt Injection 与数据外泄防护
1. 系统提示词中显式声明不可泄露系统策略、密钥、内部配置。
2. 将“用户输入”“检索内容”“系统策略”分通道处理，禁止互相覆盖优先级。
3. 对模型输出做敏感信息检测（token、密钥模式、账号模式）并阻断。
4. 引用输出仅返回允许字段，禁止返回原始内部元数据（如存储路径、内部ID映射）。

## 7.6 审计与可追踪
1. 记录每次工具调用：`session_id/user_id/tool_name/params_hash/result_meta/latency/status`。
2. 参数原文可脱敏存储，敏感字段仅保留哈希。
3. 提供审计检索接口，支持安全回放与追责。
4. 高风险调用（跨域、超量、异常模式）触发告警。

## 7.7 运行时熔断与应急开关
1. 工具级熔断：异常率超阈值自动 Open，避免持续泄露或级联故障。
2. 安全开关：
   1. 全局禁用写工具
   2. 全局禁用图谱工具
   3. 全局降级为“检索摘要模式”
3. 发现安全风险时可秒级切断相关工具链路。

1. 多模型路由策略优化（成本/效果平衡）。
